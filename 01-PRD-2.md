## System Requirements

### Technical Requirements

- **Operating Systems:** Linux, macOS, Windows (with proper UTF-8 terminal support)
- **Go Version:** 1.21 or higher
- **Terminal:** Minimum 80x24, recommended 120x30 for optimal experience
- **Disk Space:** 50MB for binary + variable project storage (avg. 10MB per novel)
- **Memory:** 50MB idle, 200MB during active use
- **Dependencies:** No external runtime dependencies (fully self-contained binary)

### Supported Terminals

- Linux: GNOME Terminal, Konsole, Alacritty, kitty
- macOS: Terminal.app, iTerm2
- Windows: Windows Terminal, ConEmu (with UTF-8 enabled)

### Minimum Hardware

- CPU: Any modern processor (Intel/AMD/ARM)
- RAM: 512MB available
- Disk: 100MB free space (for application + projects)

## Executive Summary

**syntax.sh** is a terminal-based fiction writing and worldbuilding tool. It enables authors to write stories, manage characters, organize scenes, track plots, and understand their world—all in a beautiful, distraction-free terminal interface.

### Value Proposition

- **For Authors:** Distraction-free writing with built-in narrative planning tools
- **For Worldbuilders:** Organize locations, cultures, and timelines
- **For Storytellers:** Character database, plot tracking, scene management

### Target Users

1. **Primary:** Fiction writers (literary, sci-fi, fantasy, romance)
2. **Secondary:** Screenwriters, game writers
3. **Tertiary:** Journalists, bloggers, content creators

### Positioning

> "Your novel deserves a beautiful editor. syntax.sh is built for the story first, complexity second."

---

## Product Vision

### What It Is

A markdown-first writing environment with integrated story planning tools. Write stories, track characters, organize scenes, manage locations, plot your arc, get AI assistance, and export in multiple formats—all terminal-based.

### What It Is NOT

- Not a Scrivener replacement (more focused, less comprehensive)
- Not collaborative (single-user, local files)
- Not an AI writer (AI assists, human leads)

### Kyanite Suite Integration

**syntax.sh is standalone and independent but designed to integrate with future Kyanite tools:**
- Import color palettes from prism.sh (future)
- Export writing goals to focus.sh (future)
- Use as a reference in Kyanite documentation

## Core Features

### Feature 1: Split-Pane Editor

**Purpose:** Distraction-free, beautiful writing environment

**Core Mechanics:**
- Left pane: Markdown editor with live typing
- Right pane: Live markdown preview
- Toggle full-screen editing mode
- Real-time word/line count
- Syntax highlighting for markdown
- Auto-save every 30 seconds (non-blocking)
- YAML frontmatter for story metadata

**Technical Requirements:**
- Support 10,000+ line documents smoothly
- No lag during typing
- Preview renders within 100ms
- Auto-save is non-blocking

**Success Criteria:**
- ✅ Smooth typing at any document length
- ✅ Preview stays synchronized
- ✅ No data loss on auto-save
- ✅ Works at 80x24 terminal

**Status:** Core feature, MUST ship v1.0

### Feature 2: Character Database

**Purpose:** Organize and track characters

**Core Mechanics:**
- **Create:** Name, role, description, aliases
- **Track:** Age, occupation, personality, appearance, background
- **Relationships:** Links to other characters with tension levels
- **Character Arc:** Track character development
- **Quick Ref:** Lookup character while writing
- **Relationship Map:** ASCII diagram of connections
- **Search:** Find by name, role, or attribute
- **Export:** Character profiles to markdown

**Tension Levels:**
- **Low:** Friendly, supportive relationships (allies, friends, mentors)
- **Medium:** Complex, evolving dynamics (rivals with respect, complicated family)
- **High:** Conflict, antagonistic relationships (enemies, bitter rivals)
- Used in relationship map visualization with different line styles (solid, dashed, jagged)

**Storage:**
- Location: `.story-name/characters/`
- Format: YAML metadata + markdown bio
- Auto-load when story opens

**Success Criteria:**
- ✅ Store 50+ characters without slowdown
- ✅ Search finds in <100ms
- ✅ Relationships track accurately
- ✅ Inline lookup while writing

**Status:** Core feature, MUST ship v1.0

### Feature 3: Scene Organization

**Purpose:** Structure story into hierarchical chapters and scenes

**Core Mechanics:**
- **Scene Tree:** Hierarchical structure (Acts → Chapters → Scenes)
- **Quick Navigation:** Jump between scenes with keyboard
- **Scene Metadata:**
  - POV character
  - Location
  - Time period
  - Characters present
  - Plot points
  - Status (draft/revising/done)
- **Scene Outline:** View all scenes with summaries
- **Compile:** Assemble all scenes into single document

**Compilation Rules:**
- Scenes compiled in chapter order, then scene number order
- Filter options: "all scenes", "completed only", "draft + revising + done"
- Orphaned scenes (no chapter assignment) appear in appendix with warning
- Duplicate scene numbers show warning dialog before compilation
- Missing scene numbers (gaps) are preserved (Chapter 1: Scene 1, 3, 5 is valid)

**Technical Requirements:**
- Handle 100+ scenes
- Navigate smoothly (<100ms)
- Preserve scroll position
- Deletion with confirmation

**Success Criteria:**
- ✅ Navigate 20+ scenes smoothly
- ✅ Outline shows full structure
- ✅ Compile creates valid document
- ✅ Scene metadata saved

**Status:** Core feature, MUST ship v1.0

### Feature 6: AI Story Assistant

**Purpose:** Non-intrusive AI feedback on story elements

**4 Quick Modes:**

1. **Character Check** (Ctrl+Shift+C)
   - Is character consistent?
   - Do actions align with personality?
   - Flag inconsistencies

2. **Dialogue Polish** (Ctrl+Shift+D)
   - Does dialogue sound natural?
   - Identify awkward exposition
   - Suggest improvements

3. **Scene Energy** (Ctrl+Shift+E)
   - Is scene paced well?
   - Is tension appropriate?
   - Suggest cuts or additions

4. **Plot Consistency** (Ctrl+Shift+P)
   - Does scene fit timeline?
   - Character location makes sense?
   - Flag plot holes

**Technical Specifications:**

**Model Details:**
- Model: Ollama-hosted Llama 3.2 3B (or similar 3B parameter model)
- Hosting: Local-first (Ollama), with optional API fallback
- Context Window: 2048 tokens (sufficient for scene analysis)
- Quantization: Q4_K_M (balance of speed and quality)

**API Configuration:**
- Primary: Local Ollama (http://localhost:11434)
- Fallback: Optional OpenRouter API (user-configured)
- API Keys: Stored in system keyring (macOS Keychain, Windows Credential Manager, Linux Secret Service)
- Fallback Storage: Encrypted config file (~/.config/syntax/api_keys.enc) with user warning

**Privacy & Data Handling:**
- **Local Mode (Default):** All data stays on user's machine, no external calls
- **API Mode (Optional):** User explicitly enables with warning about data transmission
- **Data Sent:** Only current scene text (max 2000 tokens), no character DB or full story
- **Data Retention:** No data stored by syntax.sh; API provider policies apply
- **User Control:** Clear toggle in settings, visible indicator when API mode active

**Offline Behavior:**
- Feature gracefully disabled when Ollama not running
- Clear message: "AI Assistant requires Ollama. Install: https://ollama.ai"
- Keyboard shortcuts show tooltip: "AI Assistant unavailable (offline)"
- No crashes or errors, just disabled state

**Performance:**
- Response time: <2 seconds (local), <5 seconds (API)
- Prompts: Under 200 tokens (scene context + instruction)
- Timeout: 10 seconds, then show "Assistant timed out" message

**Success Criteria:**
- ✅ All 4 modes work with local Ollama
- ✅ Responses under 2 seconds (local model)
- ✅ Suggestions are actionable and relevant
- ✅ No hallucinations (verified with test scenes)
- ✅ Graceful degradation when offline
- ✅ Clear privacy disclosure in help system

**Status:** Core feature, MUST ship v1.0

### Feature 8: Export & Compilation

**Purpose:** Prepare manuscript for sharing

**Export Formats:**
- **Markdown:** Single file with all scenes
- **PDF:** Formatted manuscript for printing
- **DOCX:** Agent/publisher ready (Microsoft Word format)
- **HTML:** Blog/website preview
- **Stats:** Writing statistics report (plain text)

**Technical Dependencies:**
- **PDF:** github.com/jung-kurt/gofpdf (adds ~2MB to binary)
- **DOCX:** github.com/nguyenthenguyen/docx (adds ~3MB to binary)
- **HTML:** Standard library (html/template)
- **Markdown:** Standard library (text/template)
- Total binary size increase: ~5MB for export features

**Size Limits:**
- **DOCX:** Maximum 10MB project size (~500,000 words)
- **PDF:** Maximum 50MB project size (~2,500,000 words)
- **HTML:** No practical limit
- **Markdown:** No practical limit
- Warning shown before export if limits exceeded

**Compilation Options:**
- Filter by scene status (draft/revising/done)
- Custom front matter (title page, dedication, copyright)
- Custom back matter (author bio, acknowledgments)
- Chapter break formatting (page break, ornamental separator, simple spacing)
- Appendices (character list, location index, outline summary)

**Success Criteria:**
- ✅ All formats generate error-free
- ✅ PDF properly formatted
- ✅ Scene order preserved
- ✅ Metadata included

**Status:** Core feature, MUST ship v1.0

## Data Versioning & Migration

### Schema Versioning

All story projects include a schema version in `metadata.yaml`:

```yaml
schema_version: "1.0"
created_with: "syntax.sh v1.0.0"
```

### Compatibility Policy

- **Forward Compatibility:** v1.x can read all v1.0+ projects
- **Backward Compatibility:** v1.0 projects can be opened by any v1.x release
- **Breaking Changes:** Only in major versions (v2.0, v3.0, etc.)

### Migration Strategy

**Minor Version Updates (v1.0 → v1.1):**
- Automatic migration on project open
- New fields added with sensible defaults
- No user intervention required
- Original files backed up to `.story-name/.backups/`

**Major Version Updates (v1.x → v2.0):**
- Migration tool: `syntax migrate --from v1 --to v2 <project-path>`
- Detailed migration report showing changes
- Original v1.x project preserved
- User confirmation required before migration

**Example Migration (v1.0 → v1.1 adds spell-check):**
```yaml
# Old v1.0 format
schema_version: "1.0"
title: "My Novel"

# New v1.1 format (auto-migrated)
schema_version: "1.1"
title: "My Novel"
spell_check_enabled: true  # New field, default value
spell_check_language: "en_US"  # New field, default value
```

### Data Integrity

- Checksums stored for all scene files (SHA-256)
- Validation on project open
- Corruption detection and recovery from backups
- User notification if data issues detected
